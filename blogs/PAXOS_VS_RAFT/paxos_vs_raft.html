Paxos vs Raft: The Battle of Consensus

Apr 4, 2026

When we talk about large-scale distributed systems, concepts like horizontal scaling, data replication, decoupling, and data reliability comes in the mind but at the heart it lies the idea of consensus algorithms. A consensus algorithm is a process that allows multiple nodes (servers) in a distributed system to agree on a single data value or a shared system state, ensuring the system behaves consistently as a whole. This is crucial because, in distributed environments, data is spread across different nodes, and due to network delays, failures, or simultaneous updates, inconsistencies can easily arise. Without a way to achieve agreement, different users might experience different results from the same system, leading to unreliable behavior. Consensus algorithms address this challenge by ensuring all nodes eventually agree on the same state, thereby maintaining consistency, reliability, and trust across the system.

The Replicated State Machine

To understand consensus, we have to talk about State Machines. In a distributed environment, a State Machine is a service that starts at an initial state, receives a sequence of commands, and transitions to a new state based on those commands. The goal of a consensus algorithm like Raft, Paxos and Accord is to manage a replicated log. If every node processes the same commands in the exact same order, they will all end up in the same state and in case if there is system crash from one of node in cluster other nodes will serve the data correctly.

Paxos

At its heart, Paxos is the foundational handshake that allows a messy cluster of individual computers to act as a single, reliable unit. Whether you are looking at Single Decree Paxos, Multi-Paxos, or the high-performance EPaxos(It's a leaderless approach) used in databases like Cassandra, the goal is always the same: reaching a rock-solid consensus even when the "nodes" in your network are failing or flaking out. To keep things from descending into digital chaos, the protocol operates under two strict ground rules. First is Safety, which ensures that once the group commits to a single value, it is permanently locked in—there are no "do-overs," no second-guessing, and no false alarms where a server thinks a decision was made when it actually wasn't. Second is Liveness, the functional promise that as long as a majority of your servers are awake and talking to each other, the system won't just spin its wheels; it will eventually land on a decision and ensure every other server in the pack gets the memo.

The Two Major Pitfalls: Split Votes and Conflicting Choices

Before diving into the actual flow of Paxos, it is helpful to look at two naive approaches to consensus and understand why they fail. These pitfalls perfectly illustrate why Paxos needs to be as complex as it is.

Problem 1: Split Votes (What if an Acceptor only accepts the FIRST value?)

Imagine a rule where an Acceptor is completely loyal to the very first proposal it sees and rejects everything else. If multiple Proposers speak up at the exact same time, the votes will get hopelessly split.

  • Server 1 proposes "Red", and Servers 1 & 2 accept it.
  • Server 3 proposes "Blue", and Servers 3 & 4 accept it.
  • Server 5 proposes "Green", and Server 5 accepts it.

Because no single color reached a majority (3 votes), and the Acceptors refuse to change their minds, the system is permanently deadlocked. The Takeaway: To prevent split votes, Acceptors must be allowed to change their minds and accept multiple (different) values over time.

Problem 2: Conflicting Choices (What if an Acceptor accepts EVERY value?)

To fix the split vote problem, what if we make Acceptors total people-pleasers, accepting every new value that comes their way? This introduces a catastrophic safety violation.

  • Server 1 proposes "Red". Servers 1, 2, and 3 accept it. Since that is a majority, "Red" is officially chosen.
  • Later, Server 5 wakes up and proposes "Blue". Servers 3, 4, and 5 accept it (Server 3 overwrites its previous choice). Since that is also a majority, "Blue" is officially chosen.

We now have two chosen values! This breaks the fundamental rule of consensus. The Takeaway: Once a value has been chosen by a majority, any future proposals must propose that exact same value. To enforce this, Proposers cannot just blindly shout their values; they must first "ask around" to see if a value has already been chosen. This necessitates a 2-Phase Protocol.

The Flow of Basic Paxos

To successfully agree on a single value without conflicts, Paxos breaks the negotiation down into two distinct phases. This ensures that even if multiple proposers try to talk at once, the system gracefully handles the collision based on proposal numbers.

Phase 1: The Prepare Phase (Asking for Promises)

The Proposer starts by choosing a new proposal number (let's call it n). It broadcasts a Prepare(n) message to all Acceptors.

The Acceptor receives this message. It checks its stable storage to see the highest proposal number it has ever responded to (minProposal). If the new n is higher, the Acceptor makes a promise: "I will ignore any future proposals with a number less than n." It updates its minProposal to n and replies to the Proposer. If this Acceptor had already accepted a value from a previous round, it includes that acceptedProposal and acceptedValue in its reply so the new Proposer knows about it.

Phase 2: The Accept Phase (Making the Request)

The Proposer waits until it receives promises from a majority of the Acceptors. If any of those Acceptors returned an already-accepted value, the Proposer is bound by the rules of Paxos to abandon its own value and adopt the value associated with the highest proposal number returned. If no values were returned, the Proposer can use its own original value. It then broadcasts an Accept(n, value) message to all servers.

The Acceptor receives the request. As long as it hasn't seen a higher proposal number in the meantime (meaning n ≥ minProposal), it officially accepts the value. It saves the acceptedProposal and acceptedValue to its disk (stable storage) and replies.

Once the Proposer receives successful replies from a majority of Acceptors, the value is officially chosen. If it gets rejected because an Acceptor saw a higher number from a competing Proposer, it must go back to Phase 1 and try again with an even higher number.

The Crucial Role of Proposal Numbers

As seen in the phases above, every proposal is assigned a unique number (n). The absolute golden rule here is that higher numbers take priority over lower numbers.

Because multiple Proposers might be shouting at once, a Proposer must be able to generate a new proposal number that is higher than anything it has ever seen or used before. A very simple and common approach to guarantee uniqueness across the entire distributed system is to combine a local round counter with the server's unique ID:

Round Number
Server Id

Here is how a server manages this:

  • Each server keeps track of a variable called maxRound, which represents the largest Round Number it has seen so far.
  • To generate a brand new proposal number, the server simply increments its maxRound and concatenates it with its own Server Id. (For example, if maxRound is 5 and Server Id is A, the proposal number is 5A).
  • Crucial Safety Detail: Proposers must persist their maxRound to stable storage (disk). If a server crashes and restarts, it absolutely cannot reuse an old proposal number, or it will violate the safety guarantees of Paxos!

Raft

If Paxos is so powerful and provides mathematically proven safety guarantees, why did the distributed systems world need another consensus algorithm? The answer lies in human comprehension and the harsh realities of practical software engineering.

While Paxos is elegant in theory, it is notoriously difficult to understand, and even harder to implement correctly. Furthermore, the original Paxos paper focuses heavily on "Single-Decree Paxos." To build a continuous Replicated State Machine, you have to chain these decisions together into "Multi-Paxos," which often led to complex, buggy, ad-hoc implementations.

Enter Raft. Created by Diego Ongaro and John Ousterhout, Raft was designed with one overarching goal: Understandability. It solves the "understandability gap" of Paxos by tackling the consensus puzzle through two major philosophical shifts: Problem Decomposition (breaking it into Leader Election, Log Replication, and Safety) and enforcing a Strong Leader Paradigm.

The Goal of Raft: Fault Tolerance

The ultimate goal is strict fault tolerance. If you have a cluster of 5 servers and 2 of them crash, the remaining 3 servers will still maintain the correct state machine and continue serving data seamlessly to the clients.

Terms and RPCs (The Foundation of Communication)

  • Terms: Raft divides time into logical "Terms" (acting much like Paxos proposal numbers). A Term refers to the election term. If a leader goes down, a new election is held and the term changes (i.e., term += 1). The term keeps incrementing until a new leader is successfully elected.
  • RPCs: Servers in Raft use only 2 main RPC (Remote Procedure Call) mechanisms to communicate:
    1) RequestVote (used by Candidates during elections)
    2) AppendEntries (used by the Leader for sending log data, and when empty, it acts as a heartbeat).

Server States (The Roles)

By abandoning the peer-to-peer chaos of Paxos, Raft dictates that every server must be in one of exactly three states at any given time:

Raft Server State Transitions
stateDiagram-v2 [*] --> Follower: Start Follower --> Candidate: No Heartbeat (Timeout) Candidate --> Leader: Win Election Leader --> Follower: Discover Higher Term Candidate --> Follower: Discover Higher Term Candidate --> Candidate: Timeout (New Election)
  • The Follower: The default starting state. Followers are entirely passive. They only respond to RPCs from Leaders and Candidates. If they don't receive a heartbeat in time, they transition to a Candidate.
  • The Candidate: A transitional state. The Candidate asks others for votes. If it wins a majority, it becomes the Leader.
  • The Leader: The absolute authority. The Leader accepts commands from the client, sends logs to followers, and actively sends continuous heartbeats to tell followers, "I am alive, do not start an election."

The Flow of Raft

Phase 1: Leader Election

Because all nodes start as Followers with no Leader sending heartbeats, an election timer will eventually run out. The first node to time out initiates an election.

graph TD A[Become Candidate] --> B[Current Term++
Vote for self] B --> C[Send a request to vote
to other servers] C -->|Timeout| B C -->|Votes from majority| D[Become a Leader
Send heartbeats] C -->|RPC from leader| E[Become a Follower]

Election Correctness is guaranteed by two properties:

  • Safety: Raft allows at most 1 winner per term. Each server gives only one vote per term (first-come, first-served), and a strict majority is required to win.
  • Liveness: To ensure some candidate eventually wins (and avoid split-vote stalemates), Raft chooses election timeouts randomly within a range [T, 2T] (e.g., 150-300ms). The early time-out server sends an RPC to others and asks them to vote before their own timers expire. This works exceptionally well if T is much greater than the network broadcast time.

Phase 2: Log Replication (Working Normal Operation)

Once a leader is elected, normal operation begins. The State Machine (SM) syncs via the Leader (L) in a two-step process:

  • Step I: When a user sends a command to the leader, the leader first saves that command in its own log. It then sends this data to the other nodes via an AppendEntries RPC, and those nodes also save the data into their respective logs.
  • Step II: Once safely replicated across a majority, all the nodes execute the command to their state machines. The Leader then returns a successful response to the user.

Log Consistency and Leader Completeness

Raft's strictness shines in how it handles crashes and network partitions. It enforces a Log Matching Property: the goal is a high level of consistency between logs. If log entries on different servers have the same index and term, they are guaranteed to store the exact same command, and their logs are identical for all preceding entries.

Append Entries Consistency Checks

When the Leader sends an AppendEntries RPC, it includes the <index, term> of the entry immediately preceding the new one. The follower must contain matching entries. If it doesn't match, the follower rejects the request. The leader then steps back, trying again with a lower (older) index until they find the point where their logs match, and overwrites the follower's log from there.

Example 1 (Clean Accept):
Leader Log: [Term 1] [Term 1] [Term 2] [Term 3]
Follower Log: [Term 1] [Term 1] [Term 2]
Result: Follower accepts the next entry because index 3 (Term 2) perfectly matches.

Example 2 (Reject & Rewrite):
Leader Log: [Term 1] [Term 2] [Term 3] [Term 4]
Follower Log: [Term 1] [Term 1] [Term 1] [Term 1] [Term 1] [Term 1] [Term 1]
Result: The follower rejects the new append because index 3 doesn't match. To maintain consistency, the Leader forces the Follower to overwrite its log entirely to match the Leader's state.

Leader Completeness (Safety Check)

How do we know a newly elected leader has a complete log? Servers with incomplete logs must not be elected.

  • Once a log entry is committed by a majority, all future leaders must store that entry.
  • To enforce this, a Candidate must include the index and term of its last log entry in its RequestVote RPC.
  • When a server receives a vote request, it checks its own log. It will deny its vote if its own log is more "up-to-date" than the candidate's.
  • Logs are ranked dynamically by checking <lastTerm, lastIndex>.
